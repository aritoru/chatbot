# AD&DHelp Conversational Agent

An AI-powered chatbot that interviews customers about their AD&D questions, collects structured data about their problem, and answers AD&D knowledge questions on demand.

## Language

**Customer**:
A person interacting with the Agent to get help with an AD&D question. Always anonymous — no login or identity required.
_Avoid_: User, client

**Agent**:
The AI system that drives the Interview and answers AD&D knowledge questions.
_Avoid_: Bot, chatbot, assistant

**Interview**:
The Agent-driven flow that collects four fields from a Customer: game system, problem category, problem description, and urgency level.
_Avoid_: Conversation, session, form

**Interview State**:
The Agent's internal record of which Interview fields have been collected and which question is currently pending.
_Avoid_: Context, memory, session state

**Game System**:
A constrained field identifying the AD&D edition: AD&D 1e, AD&D 2e, D&D 3/3.5, D&D 4e, D&D 5e, or Other. The Agent re-asks if the Customer's response doesn't map to one of these.
_Avoid_: Edition, version, ruleset

**Problem Category**:
A free-text field describing the type of AD&D problem the Customer has (e.g., rules clarification, combat, spellcasting). Not validated against a fixed list.
_Avoid_: Issue type, tag, topic

**Problem Description**:
A free-text field where the Customer describes their AD&D problem in their own words.
_Avoid_: Details, notes, body

**Urgency Level**:
A constrained field with exactly three valid values: Low, Medium, or High. The Agent re-asks if the Customer's response doesn't map to one of these.
_Avoid_: Priority, severity

**Confirmation Step**:
The final stage of an Interview where the Agent presents a summary of all collected fields and asks the Customer to confirm before the Intake is created. On rejection, the Agent corrects one field at a time and re-presents the summary — it does not restart the Interview.
_Avoid_: Review, approval, sign-off

**Intake**:
The structured JSON record produced at the end of a successful Interview — contains the five collected fields, a generated summary, the Issue Reference Number, a timestamp, and the full conversation transcript.
_Avoid_: Ticket, report, form

**Issue Reference Number**:
A unique identifier generated by the system when an Intake is created and shown to the Customer at the end of the Interview. The Customer never provides it.
_Avoid_: Ticket ID, case number, session ID

**Frustration Signal**:
The Agent's internal classification of the Customer's emotional state during an Interview, inferred from tone, word choice, and message length on every Customer turn. Values: `None`, `Mild`, `High`. Never disclosed to the Customer in conversation. Recorded on the Intake for support staff visibility.
_Avoid_: Sentiment, frustration level, mood score

**Knowledge Query**:
An AD&D question asked by a Customer mid-Interview that the Agent answers before resuming the Interview. Does not modify Interview State.
_Avoid_: Tangent, side question, off-topic

**Session**:
A server-side in-memory record keyed by a UUID, holding the conversation history and Interview State for one Customer interaction. Cleaned up when the Interview completes or is explicitly closed.
_Avoid_: Connection, thread, user session

## Relationships

- A **Customer** participates in one **Interview** per **Session**
- An **Interview** produces exactly one **Intake** upon successful completion of the **Confirmation Step**
- An **Intake** has exactly one **Issue Reference Number**, assigned at creation time
- A **Knowledge Query** does not modify **Interview State** — the Agent answers and re-asks the pending question
- A **Knowledge Query** can occur zero or more times during an **Interview** without terminating it
- A **Session** is destroyed when the **Interview** completes or is explicitly closed

## Example dialogue

> **Dev:** "If the Customer asks 'what's THAC0?' mid-interview, does that pause the Interview?"
> **Domain expert:** "No — that's a Knowledge Query. The Agent answers it, then re-asks the pending Interview question before waiting for input."
>
> **Dev:** "When does the Customer learn their Issue Reference Number?"
> **Domain expert:** "At the end of the Confirmation Step — once the Intake is saved, the Agent tells the Customer their number."
>
> **Dev:** "What if the Customer says the urgency is wrong after seeing the summary?"
> **Domain expert:** "The Agent asks which field to correct, takes the new value, and re-presents the summary. It doesn't restart."
